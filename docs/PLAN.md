# Lab Automation Tycoon — 設計計画書

研究者の研究活動・研究生活をリアルに表現したタイクーン系シミュレーションゲーム。
ゼロから設計し、データ駆動・ドメイン/UI分離で長期的な拡張を担保する。

---

## 1. ゲームコンセプト

### 1.1 一行コンセプト
> 大学院生で始め、研究と雑務に追われながら、グラントで自動化されたラボを築き、トップ研究者を目指す日次ターン制タイクーン。

### 1.2 体験のコア (Three Pillars)
- **リソース配分の苦悩** — 1日の時間/集中力は有限。実験・執筆・指導・雑務のどれを削るか毎日決断する。
- **長期投資のリターン** — 機器導入や学生育成は今日は痛い、半年後に効いてくる。短期と長期の駆け引き。
- **自動化による解放** — 序盤は手作業地獄、中盤から機器/学生/SOPに任せ、終盤は戦略レイヤーに専念する**ステージ感**。

### 1.3 ターゲットなプレイ感
Two Point Hospital のキャラ運営感 + Universal Paperclips の数値最適化感 + Academia: School Simulator の運営感 + 研究者あるあるの**ブラックユーモア**。

## 2. コアゲームループ (1ターン = 1日)

```
朝フェーズ:    今日の予定を組む (時間・集中力の割当)
        ↓
昼フェーズ:    アクション実行 (実験/執筆/指導/雑務) + ランダムイベント
        ↓
夜フェーズ:    SOP・学生による自動進捗、結果サマリー、ログ蓄積
        ↓
週次/月次:    給料、機器メンテ、グラント締切、学会、査読、共同研究貢献
        ↓
学期/年次:    昇進判定、学生卒業、研究室評価、長期目標進捗
```

**毎ターンの「やりたいこと > 出来ること」のテンション**が楽しさの源泉。

## 3. リソース・ステータス

### 3.1 1日のリソース
| リソース | 概要 |
|---|---|
| **時間 (Time Slots)** | 1日のアクション枠 (graduate=3, postdoc=4, ...) |
| **集中力 (Focus)** | スロット消費で減衰。低いとアクション成功率↓ |
| **メンタル (Mental)** | 締切・トラブルで減少。低いとバーンアウトリスク |

### 3.2 永続リソース
| リソース | 概要 |
|---|---|
| **研究資金 ($)** | 試薬・機器・人件費・出張費。グラントで補充 |
| **研究ポイント (RP)** | 内部経済の軸、スキルツリー解放や論文の質に寄与 |
| **評判 (Reputation)** | 学会・論文・グラント採択で増。コラボ依頼・査読依頼の引きに影響 |
| **業績 (Publications / H-Index)** | 査読付き論文の累積。長期評価軸 |
| **キャリアレベル (Career Tier)** | 院生→ポスドク→助教→准教→教授→拠点リーダー |

### 3.3 個人スキル (XP制)
実験技術、データ解析、執筆、英語、プレゼン、マネジメント、ラボ運営。
アクションで対応スキルにXPが入り、レベルアップで効率/成功率/メンタル消費軽減。

## 4. ゲーム内システム

### 4.1 実験 (中核)
プロトコル = 必要試薬・機器・スキル・時間スロット・成功確率。成功で品質/再現性スコアを持つデータポイントを獲得。失敗もスキルXPと経験になる。ウェット (手作業多・機器自動化効く) と ドライ (計算ボトルネック・GPU効く) の両系統。

### 4.2 機器・設備
グレード/コンディション/スペース/維持費。`automation: manual | semi-auto | auto` を持ち、auto機器は SOP に紐付けて自動運転可能。後から実機材 (PCR/シーケンサ/GPUクラスタ/Opentrons/Biomek) をデータ追加だけで増やせる。

### 4.3 プロジェクト
1プロジェクト = 複数実験 + 1〜数本の論文化。`planning → preliminary → main → analysis → writing → submitted → 受理` のステージをデータ蓄積に応じ自動進行。複数並行管理可。

### 4.4 論文・出版
論文品質 = データ品質平均 × 執筆スキル。ジャーナル選択 (低IF確実 〜 高IFハイリスク)、査読 (Major/Minor Revision)、再投稿。受理で業績+評判+RP獲得。

### 4.5 グラント (申請書)
小型 (学内若手) 〜 大型 (CREST/NIH R01)。執筆ポイント蓄積→提出→28日後採否判定。評判/論文数で採択率補正。採択で大型資金獲得→機器/人員拡張。

### 4.6 学生・スタッフ
B4〜ポスドクまで。スキル/独立性/モチベ/メンタル/専門タグを持つ。1on1指導でスキル&独立性UP。**毎ターン終わりに自動実験を回す** (独立性が低いと失敗多発)。給与は月次。

### 4.7 学会・コミュニティ
学内ワークショップ 〜 国際ゴードン会議。申込→14日後自動会期到来→評判+RP獲得。ポスター/口頭/招待で倍率変動、国際は英語スキル要。

### 4.8 共同研究
Collaborator (京大山田研/MIT/産総研/EU 等) と契約 (60〜120日)。毎ターン RP+データ品質を貢献。完了時に評判+資金+共著論文品質ボーナス。

### 4.9 キャリア昇進
6段階。昇進条件 = 業績点 + 評判 + H-Index + 採択グラント数。昇進で時間スロット/学生定員/ラボスペース変動・雑務率増 (苦いトレードオフ)。

### 4.10 ランダムイベント
日次抽選 (条件述語 + 重み付き)。機器故障/招待講演/スクープ/査読依頼/学生不調/雑務/取材/停電/コンタミ/委員会… プレイヤーは選択肢付きモーダルで応答。

### 4.11 機器自動運転 (SOP) — このゲームの目玉
semi-auto/auto機器に「機器インスタンス × プロトコル」のペアを SOP として登録。**プレイヤー不要で毎ターン自動実行**。ロボット機器は `parallelSlots` で1機器に複数SOP可。試薬消費40%削減・コンディション消耗。

### 4.12 機器修理サイクル
SOP/学生実行で機器コンディションが下がる。低いと故障率↑、ランダムイベント「機器故障」発生確率↑。「修理」アクションで100に回復 ($と時間)。

## 5. 進行・ステージ感

1. **修行期 (院生〜ポスドク前半)** — 手作業の実験、雑用、メンタル耐久。
2. **独り立ち期 (ポスドク後半〜助教)** — 自分のプロジェクト、学生1人めの指導。
3. **拡張期 (准教授)** — 学生複数、グラント獲得、機器の自動化開始。
4. **戦略期 (教授〜)** — ラボの方向性決定、後進育成、コミュニティ貢献。手作業ほぼゼロ。

## 6. 技術スタック

**Web (TypeScript必須) + React + Zustand**

| 層 | 採用 | 理由 |
|---|---|---|
| 言語 | **TypeScript** | データ駆動の型安全に必須 |
| ビルド | Vite | 起動秒速、HMR、ゼロ設定でTS対応 |
| UI | **React 18** | tycoon系の「UI/メニュー/データテーブル中心」に最適 |
| 状態管理 | **Zustand** (+ persist + immer) | Reduxより軽量 |
| スタイル | Tailwind CSS | 反復改修の生産性 |
| テスト | Vitest + Testing Library | Vite統合 |
| 保存 | localStorage (zustand persist) | バージョン管理+マイグレーション |

**React vs Phaser**: 日次ターン制 + メニュー主体なら React 圧勝。将来ラボ可視化が必要なら Pixi.js を局所採用 (Phaser不採用)。

## 7. アーキテクチャ (3層)

```
┌─────────────────────────────────────┐
│ UI Layer (React)                    │ ← layout/, presenters/
└──────────────┬──────────────────────┘
┌──────────────▼──────────────────────┐
│ State Layer (Zustand)               │ ← store/gameStore.ts
└──────────────┬──────────────────────┘
┌──────────────▼──────────────────────┐
│ Domain Layer (Pure TypeScript)      │ ← React/DOM非依存・Nodeでテスト可
│  - entities/types  data/  actions/  │
│  - engine/  rules/                  │
└─────────────────────────────────────┘
```

- **ドメイン層は React/DOM 完全非依存**。
- 状態変更は純粋関数 `(prevState, input) => newState` (Immerで簡潔に)。
- Zustand はドメイン関数を呼ぶアダプタに徹する。
- UIはドメインを直接触らず、`ui/presenters/selectors.ts` を経由した ViewModel を介する。

## 8. データ駆動設計

全コンテンツを `src/domain/data/` 配下の TS ファイルに定義 (JSONより型補完が効く)。

```
data/
  protocols.ts    grants.ts       events.ts
  equipments.ts   journals.ts     conferences.ts
  students.ts     collaborators.ts careers.ts
```

**新コンテンツ追加 = データ配列に push するだけ**。ルール側は汎用関数で解釈。

## 9. UI方針 (短期 → 長期)

### 現状の構成 (P6.5 完了時点)
```
┌─ HUD ──────────────────────[📋 進行]┐ ← モバイル時はトグル
├─ SceneArea ──────────────────[🧪][💻]│ ← SceneNav (移動/終了)
│   背景 + キャラ立ち絵        [🎤][🏠]│   オーバーレイ右端
│                              [🌙]   │
├─ NarrationArea ─────────────────────┤
├─ ActionPanel (シーン依存タブ・高さ固定)┤
└──────────────────────────────────────┘
                    ↑
            TaskPanel (進行ボード)
            md+: 常時サイドバー
            mobile: ドロワー (スライドオーバー)
```

中身はまだテキスト+絵文字、レイアウト骨格は完成。

### 長期: ソシャゲ風アセット差し替え (P7)
SceneArea を背景画像 + キャラ立ち絵に、ActionPanel をカードUIに差し替える。

### 差し替え戦略 (現実装済)
1. **「シーン」概念**を `gameState.scene` で持つ → 背景/立ち絵を UI 側で解決
2. **キャラはView-Model駆動** (`selectPresentCharacters`) → ドメインは誰がいるかだけ持つ
3. **`ActionDescriptor` で抽象化** → テキストボタンでもカードUIでも描ける
4. **layout/ と presenters/ の差し替えだけ**でロジック非影響
5. **UI状態は uiStore.ts に分離** → ゲーム本体stateと混ぜない

## 12. 落とし穴と対策

| 落とし穴 | 対策 |
|---|---|
| ドメイン肥大化 | エンティティ単位でファイル分割、変更は専用reducer経由 |
| データ散らかり | `data/` 配下に集約、型補完で安全 |
| イベント爆発 | 「条件述語 + 効果リスト」のシンプル構造に統一 |
| セーブデータ互換破壊 | バージョン番号 + マイグレーション関数 (現在 v4) |
| UIだけ進んでロジック追いつかず | ドメイン層から作る |
| 「リアルだけど苦痛」になり面白さ低下 | リアル要素にも必ず**笑える/嬉しい/解放感**を伴う見せ方 |
| 自動化が早すぎて中盤空洞化 | 自動化解禁は中盤以降、序盤は手作業の意味を成立させる難度 |
